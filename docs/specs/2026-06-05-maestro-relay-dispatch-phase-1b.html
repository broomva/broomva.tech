Relay-Dispatch Runtime

Turn a queued dispatch into a real agent running on a live relay node — and drive orchestration state from real run events. Both ends.

00Why this exists

Today, clicking Trigger in Maestro is control-plane bookkeeping with no execution behind it.

The shipped slice (BRO-1367/BRO-1393) records a queued specDocRun, spends the N=1 dispatch budget, and advances orch_state proposed → triggered — in one advisory-locked transaction. But nothing consumes that queued run. Verified: the only writers of specDocRun are the schema and triggerSpecDoc() (which inserts the queued row) — no reader exists. No dispatcher, no runner. So a triggered spec parks forever, and (because triggerSpecDoc is the sole orch_state writer) there's no un-trigger.

BRO-1408 hid the Trigger button to remove that foot-gun. This spec is the other half: the runtime that makes Trigger mean something. The user's framing was exact — "that still needs work on both ends."

03The two ends

End A — Server dispatcher broomva.tech

A worker that closes the loop between a queued run and a live session. Responsibilities:

  • Claim the next queued specDocRun (owner-scoped), advisory-locked so two workers never double-spawn.
  • Select a node — an online relayNode owned by run.ownerId (status='online', fresh lastSeenAt). If none: leave queued, surface "no online node" (don't burn the attempt).
  • Spawn — reuse the verified path (insert relaySession + rPush spawn cmd to nodeCommandsChannel), but carry the new fields (§05). Seed the prompt with the same continuePrompt(doc) the Continue button uses (app/maestro/lib.ts) — single source of truth.
  • Advance — on 201 spawning: run.status=running, run.runRef=relaySession.id, orch_state → running.
  • Consume events — react to runner write-back (§05) to advance running → review|blocked, idempotent on run.lastSeq.

Mechanism (open — §07): a cron/queue worker polling queued rows, or event-driven off the trigger transaction. Start with a small polled worker (every ~5s) — simplest, and the volume is N=1-per-spec-version.

End B — Runner write-back relayd + agent

The relay daemon already polls /api/relay/poll and spawns processes. What's missing for Maestro:

  • Accept the seed — the spawn config must carry the prompt (or specHandle for the runner to broomva docs get) + specDocRunId. The daemon starts the claude-code process pre-fed that prompt.
  • Emit a receipt — on agent completion, POST a typed receipt event to /api/relay/events referencing specDocRunId, carrying evidence (PR URL / commit / test summary / artifact). This is what makes Review evidence-gated rather than vibes.
  • Heartbeat the run — optional progress events (lastSeq) so the board can show liveness.
Security boundary (must resolve before shipping). The runner executes an agent against an arbitrary owner-authored prompt with repo access on a real node. This is remote code execution by design. Decide the sandbox / allowed-workdir / approval-gate policy (§07) before the dispatcher is allowed to spawn from a web click. Today the human runs Continue/Copy themselves — that human-in-the-loop is the current safety; the dispatcher removes it.

05Contract additions

Two additive changes — keep them backward-compatible (the relay path serves the existing console too).

1 · Spawn config gains Maestro fields

config: {
  name, workdir, model?, sessionId,        // existing
  specDocRunId: string,                     // correlate write-back → run
  seedPrompt: string,                       // = continuePrompt(doc), pre-fed to the agent
  // or, instead of seedPrompt: specHandle so the runner fetches via `broomva docs get`
}

2 · A typed receipt event (runner → /api/relay/events)

{ type: "receipt",
  specDocRunId: string,
  outcome: "completed" | "failed" | "needs_approval",
  evidence: {
    prUrl?: string, commit?: string,
    testsSummary?: string, artifactRef?: string, notes?: string
  },
  seq: number }                              // idempotency vs run.lastSeq

The dispatcher maps outcome → orch_state: completed → review (with run.receipt=evidence), failed → blocked (retry if budget), needs_approval → blocked (human resumes via existing sessions/[id]/approve).

06Two locked decisions this inherits

D9 · Evidence-gated Review

A prose-only human "accept" launders unverified completion into an audit trail. review → done is allowed only when run.receipt is present and its evidence resolves (PR merged / tests referenced / artifact fetchable). The accept endpoint checks the receipt, not the human's say-so.

D5 · Dispatch ≠ attempt

Dispatch identity is N=1 per immutable spec-version (dispatchCount + unique idempotencyKey). Execution attempts are bounded retry (attempt/maxAttempts on the run). A blocked run retries without minting a second dispatch — same run, next attempt.

09Validation (P11) — the acceptance test

  1. Ensure one online relay node owned by you (GET /api/relay/nodes → .nodes, not a bare array).
  2. Publish a trivial spec; Trigger it → assert orch_state triggered then running within the dispatcher interval; assert a real relaySession exists + a claude-code process started on the node (node logs).
  3. Let the agent finish a tiny task (e.g. open a draft PR); assert a receipt event arrived and state is review with run.receipt.prUrl set.
  4. Try to accept with the receipt removed → blocked (D9 holds). Restore → accept → done.
  5. Re-trigger the same version → 409 (N=1 + idempotency). Force a failure mid-run → blocked → retry increments attempt, not dispatchCount (D5).
