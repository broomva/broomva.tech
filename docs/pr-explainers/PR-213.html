broomva docs publish

Finish an HTML spec → get an instant, phone-openable, owner-gated URL.

The problem. The autonomous workflow emits Category-C HTML artifacts (specs, PRDs, architecture). On a remote session the agent hands back "the spec is at /Users/…/docs/specs/foo.html" — a local path that can't be opened from a phone. Now:

broomva docs publish spec.html  →  https://broomva.tech/d/<id>

The flow

Why doc-as-data, not "PR → deploy → view"

Vercel preview deploys (*.vercel.app) can't see the production broomva.tech Neon Auth cookie — cookies are domain-scoped — so per-PR-preview gating breaks, and a PR+deploy costs minutes. The HTML is stored server-side and served at a stable apex-domain route: live in seconds, and the phone is already logged in.

Gating — the authenticated identity, nothing hardcoded

Ownership is doc.ownerId === resolveAuth(req).userId. The CLI Bearer Life-JWT's sub and the browser session's user.id resolve to the same user.id (device/authorize → signLifeJWT({id}) → verifyLifeJWT). Every read/list/delete is owner-scoped (and(eq(id), eq(ownerId))); a non-owner gets 404 — no existence leak. The document HTML renders in a sandboxed <iframe srcDoc> (allow-scripts without allow-same-origin): its inline JS and charts run, but it cannot read the broomva.tech session cookie, touch the parent DOM, or make credentialed same-origin requests.
